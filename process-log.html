Tue, Mar 3, 2026
Debugging in the dark, then the lights came on
Spent most of the day chasing a bug that turned out to be three bugs stacked on top of each other — and in doing so, stumbled into a completely different way of working. By the end of the day, the chart was showing real data for the first time, the tool had been handed off to Claude Code, and the candidate page could load any politician in the country, not just MGP.
–Spent the morning manually inspecting Network tab responses and pasting API output into Chat — a slow but instructive loop that eventually pinpointed why filing deadline markers weren't showing up
–Discovered the FEC API silently ignores unrecognized query parameters — the date filters we'd been passing for weeks were doing nothing; the fix required a completely different approach to how we query for deadlines
–Installed Claude Code, set up a local dev server, and moved the build workflow out of Chat and into the terminal — a significant shift in how the project gets built day to day
–Added a CLAUDE.md file to the repo — a persistent brief that orients Claude Code at the start of every session, capturing decisions, field names, and context that would otherwise be lost between conversations
–Removed all hardcoded candidate references; the page now works for any candidate by passing ?id= in the URL — tested against both a House and Senate candidate
–Chart x-axis now spans the full election cycle for any office type — a Senate candidate's chart correctly renders a 6-year window
On silent failures
The FEC API doesn't error when you pass an unrecognized query parameter — it just ignores it and returns everything. The deadline filter looked like it was working because the results accidentally landed in the right date range. The lesson: always verify a filter is actually filtering by checking the total result count, not just the shape of the returned data. Claude Code made this kind of verification instant — collapsing a debugging loop that had taken several Chat rounds into a single terminal command.
On switching tools mid-project
Moving from Chat to Claude Code mid-session felt like switching from whiteboard to workshop. Chat was the right place to think through decisions, learn how the tools work, and build the brief that would carry context forward. Claude Code is the right place to build — it can read files, run commands, and verify API responses autonomously. The two aren't competing; they're different modes of the same collaboration.
Field notes
Debugging isn't always about the code — sometimes it's about understanding what the API is actually doing versus what you assumed it was doing. Learned to verify filters by checking result counts, not just eyeballing the data shape. Also took a first real step into agentic AI tooling: not just prompting, but setting up a persistent context file so an AI tool can orient itself in a codebase independently. That's a different kind of skill than most designers are building right now.
Field notes
"What does financially healthy actually mean to a strategist?" is a design question, not a data question — and the answer changes what you build entirely. Also learned that working with an AI collaborator on a codebase requires the same thing good design requires: knowing what problem you're actually solving before you start making things.

Wed, Mar 4, 2026
The data was there — I was just asking the wrong question
Two tabs shipped today: Raised and Spent. Both required the same move — recognizing that the first technically correct version was answering a less interesting question than the data actually supported. The day started with a donor table that worked fine and ended with a spending breakdown built on top of a free-text field the FEC doesn't enforce. Everything in between was about finding the right question.
–Raised tab shipped: contributor type donut, contributor geography choropleth (D3-geo + TopoJSON), and top committee contributors table
–Mid-session pivot: replaced Top Individual Donors with Top Committee Contributors — individual contribution limits make the individual list too compressed to be meaningful; one parameter flip on the Schedule A call (is_individual: false) surfaced the more interesting question
–Added diagonal stripe pattern to the unitemized donut segment to communicate "same source, less granular" — implemented via native Canvas API, no plugin needed; Chart.js 4 accepts CanvasPattern objects natively
–Dedup logic for committee contributors uses FEC committee ID rather than name-matching — more reliable since the same PAC can file under slight name variations
–Spent tab built from scratch: spending by category donut (FEC totals endpoint), spending by purpose horizontal bar chart (keyword-mapped), and top vendors table
–Discovered disbursement_purpose_description is always null in FEC data — actual text lives in disbursement_description; fixed across all spend logic
–Built keyword-mapping layer on top of FEC's four coarse spend buckets — surfaces categories like TV & Radio, Digital, Direct Mail, Staff & Payroll, Consulting, Events & Travel, Legal & Compliance; methodology disclosed transparently above the chart
–Implemented cursor-based pagination for Schedule B — ensures full coverage for any candidate (MGP's 2024 cycle had 2,227 disbursements)
–Added global disclaimer banner: "Early build, actively evolving. All data sourced from the FEC public API. Non-partisan by design."
–Deployed to Netlify
On finding the right question
The FEC's own spend categories sound useful until you look at them: ADVERTISING, MATERIALS, TRANSFERS, OTHER. "OTHER" contains payroll, digital consulting, legal fees, and bank charges — all lumped together. The granularity is there, it just lives in a free-text field nobody enforces. The individual donor table had the same problem from the other direction: technically correct, but individual contribution limits make the amounts too compressed to tell you anything strategic. Both fixes required the same thing — noticing the question was wrong before touching the code. That's the part that takes the most time and shows up least in a changelog.
Field notes
What you get out of AI is a reflection of what you bring to it — your values, your instincts, your priorities. The pivot from individual donors to committee contributors wasn't a technical decision; it was a product instinct about what campaign finance data is actually for. The code change was one parameter. The judgment call behind it took five days of learning this domain to make. Designing in this environment is less about knowing how to use the tools and more about knowing what to ask them to build. That same principle showed up in how the day was structured: opening a Claude Code session with a clear context file and a deliberate prompt, closing it with a drafted log entry. Those aren't just workflow habits — they're the skill. CLAUDE.md isn't a readme for the codebase; it's a brief for a collaborator who forgets everything between sessions. Knowing how to write it, keep it current, and hand off between tools is its own kind of design work. The workflow itself kept getting more fluent as the day went on. Plan mode used strategically at the start of a task — review, push back, approve — rather than as a default. Auto-accepting edits running in the background while attention goes to strategy, domain questions, UX. Steering mid-task instead of waiting for completion. Five days ago this felt like learning a new tool. Today it felt more like working with a partner — one where understanding how it thinks is what makes the collaboration actually productive.
